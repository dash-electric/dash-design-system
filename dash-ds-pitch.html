02 · Symptoms
What the drift looks like in the codebase today

Six different definitions
of the same idea.

Symptom 01

Six "Primary Button"s

Each app re-implements the primary action button. Purple shifts by hue, padding by 2–4px, radius from 4 to 12. None reference a token.

Symptom 02

Forms invented per feature

Every team picks a form library (RHF, Formik, raw). Validation message styles diverge. Error semantics are inconsistent across screens.

Symptom 03

Brand purple in 11 hex values

Grep shows #5E2AAC, #5E2BAC, #5E29AB, #5D2AAC — all "Dash Purple", none of them the canonical one.

Symptom 04

Light/dark theming is per-app

Some surfaces ship dark mode, some don't, some get it half-right. There is no shared semantic layer that says "what surface is this".

Symptom 05

Reviews argue spacing, not logic

The most expensive minutes of a senior engineer are spent saying "8 not 10" in a comment thread. That cost is paid every week.

Symptom 06

New surfaces start from scratch

A new dashboard means rebuilding cards, tables, charts. The library that should accelerate this is the one we don't have.

03 · Hypothesis
The proposal

One layered system
every team installs, none can silently fork.

A design system is not a component library. It is a contract: a finite vocabulary of tokens, a versioned set of primitives, a semantic theme layer, and a small catalogue of composed blocks — installed via CLI into every consuming repo, additive only, audited on every change.

What it is

  • A package suite — installed into each app, versioned, ejectable but never silently mutated.
  • A token contract — color, type, space, radius, shadow, motion. One source, one spelling.
  • An additive surface — teams can extend, never replace. Hostile forks fail review.
  • An audit trail — every change goes through the registry, every consumer sees the diff before adopting.

What it is not

  • Not a UI framework — we are not asking teams to throw out React, Next, or their app shell.
  • Not a website — the docs surface is a consequence, not the artefact. The contract lives in code.
  • Not a brand refresh — visual identity stays. We are encoding what exists into a system that holds.
  • Not optional per-app — once we choose this, opting out is a Head-level decision, not an engineer's.

05 · Foundation
The alphabet

If it has a value,
it has a name.

Foundation is a finite, versioned dictionary. Every higher layer references it by name, never by literal. Change the dictionary, and every downstream surface updates atomically.

// packages/foundation/tokens/color.ts export const color = { brand: { 50: '#F4EDFB', 100: '#EEE7FA', 500: '#5E2AAC', // canonical 700: '#3A1670', 900: '#1F0840', }, ink: { 900: '#0E0E10', 600: '#4A4A52', … }, surface: { paper: '#F7F7F5', card: '#FFFFFF', … }, }
C

Color

50 / 100 / 500 / 700 / 900 ramps per role. Light + dark resolved at the theme layer, never inline.

14 ramps · 70 stops
T

Typography

One display face, one body face, one mono. 8 sizes, 4 weights, locked line-heights.

3 families · 32 styles
S

Space & Radius

4px base grid. Radii from sharp (2px) to pill (999px) on a fixed scale. No off-grid values.

8 steps · 6 radii
M

Shadow & Motion

5 elevation steps, 4 motion curves, 3 duration tiers. Enough to express, finite enough to govern.

5 elevs · 12 curves

07 · Primitives
Layer 1 · headless, accessible, behaviour-only

Atoms that already know
how to keyboard, focus, and announce.

Primitives carry no brand. They carry interaction grammar — focus management, keyboard navigation, ARIA semantics, ID wiring. They are the parts of UI that should never be re-implemented per feature.

Action

Button

Loading, disabled, icon, size variants. Forwards refs, polymorphic via asChild.

Input

Field

Label, hint, error, success — all derived from one state prop. No invented forms.

Overlay

Dialog · Popover · Menu

Focus trap, dismiss on escape, restore focus on close. Behaviour solved once.

Navigation

Tabs · Breadcrumb

Roving tabindex, route-aware variants, server-component-safe.

Feedback

Toast · Alert · Banner

Polite vs assertive announcements wired correctly. Promise variants for async work.

Data

Table · List · Tree

Selection, sorting, expansion — headless. Themed at Layer 3.

Media

Avatar · Image · Icon

Fallback semantics, lazy loading, sizes presets honoured.

Form

FormProvider

Validation contract is library-agnostic. Bring zod, valibot, or none.

08 · Themes
Layer 2 · semantic roles, not literal values

Components don't ask
"what colour?". They ask
"what role?"

Roles, not paint

A button doesn't say color: #5E2AAC. It says color: surface.brand. The theme decides what that resolves to, in this product, on this surface, in this mode.

// packages/theme/dash-light.ts export const dashLight = { surface: { paper: token.color.surface.paper, card: token.color.surface.card, brand: token.color.brand[500], muted: token.color.ink[100], destructive: token.color.danger[500], }, ink: { default: token.color.ink[900], muted: token.color.ink[600], onBrand: '#FFFFFF', }, }

Why this matters

  • Dark mode is one file. Re-bind the roles. Components don't change.
  • Sub-brands are one file. Same primitives, different theme.
  • A/B brand tests are one toggle. Wrap a route in a different ThemeProvider.
  • The brand purple is canonical once. Change it in foundation; every screen updates atomically.
  • Accessibility ratios are guarded. The contract enforces minimum contrast at theme-resolve time.

10 · Consumption
From registry to repo · five steps

A team adds a block in one command,
and the diff lands in a PR.

01
Search

Engineer asks the CLI or MCP server: "do we already have a payout card?"

02
Add

dash add payout-release copies the block into the consuming repo. No runtime dep.

03
Compose

The block uses primitives and theme tokens. Brand is inherited, not re-spelled.

04
Validate

Pre-commit gate runs the registry validator — token-only colors, no banned imports, audit fields present.

05
Ship

PR opens with the diff against the canonical version. Drift is visible at review time.

# in any app repo $ dash add payout-release → resolving against registry… → pulling block payout-release@1.2.0 → writing src/components/payout-release/* → wiring @dash/theme · @dash/primitives → running validator… PASS (foundation: 100/100) ✔ Block added. Review the diff and open your PR.

Three consumption surfaces

  • CLI (dash) — engineers, in-terminal, day-to-day.
  • MCP server — AI assistants (Claude, Cursor, Copilot) discover and use the registry natively.
  • Skill (v3) — repo-local rules injected into the AI context so generated UI is system-compliant by construction.

11 · The builder on top
The system, as a builder anyone can use

A browser, a prompt,
and a pull request
at the other end.

Once the system is real, the leverage compounds. A team member — engineer or not — describes a screen in their own words. The builder uses the design system, the skill rules, and the user's own Claude subscription to generate compliant code and open a PR. No API key paid by the company. No drift introduced.

1

Install once

npm i -g @dash/build && dash-build. A local daemon starts; the browser opens to a dashboard.

2

Sign in

Anthropic OAuth against the user's own Pro/Team subscription. GitHub App for repo access. No keys stored.

3

Prompt

"Add a payout export filter to backoffice." The builder asks clarifying questions before generating anything.

4

Generate

Skill chain injects PRD framing → layered architecture → repo-local rules → design tokens. Output is validated, not free-form.

5

Preview

Sandboxed iframe renders the generated TSX. The foundation match score (0–100) is visible before anything is committed.

6

Open PR

One click. Files are written, the branch pushed, a PR opened against the target repo, ready for review.

13 · Adoption
How this lands across the org, in waves

No forced migration.
Pull, not push.

Adoption is a function of leverage, not mandate. Each wave makes the next one easier. By the time a team is asked to migrate, the system already saves them more than it costs.

Wave 1–2 · done

Foundation + Primitives

Packaged, published, documented. Used internally on the design system itself.

Wave 3 · done

Theme + CLI

Consumption story sealed. First external app (backoffice) starts pulling blocks.

Wave 4 · in flight

Pilot apps

Backoffice, Mitra mobile, Halo Dash. New surfaces on the system; old surfaces ported opportunistically.

Wave 5 · this week

Builder pilot

3 internal users, mixed roles. Validates that non-engineers can ship UI without breaking the system.

Wave 6 · next month

BU-wide invites

Each BU appoints a system champion. Office hours, weekly diff review.

Wave 7

Migration of remaining surfaces

Driven by champions, prioritised by which screens take the most review time today.

Wave 8

Quarterly cadence

Versioned releases. Public changelog. RFC process for foundation changes.

Always-on

Drift detection in CI

Every PR in every repo runs the validator. Drift becomes a review comment, not a long-term debt.

14 · Risks
Honest risks, with mitigations

What could go wrong.
And what we do about it.

Risk
Teams treat the system as optional

If adoption is voluntary forever, drift continues silently in parallel repos.

Mitigation
CI drift detection from day one

Every PR runs the validator. Drift is a comment, not a quiet decision.

Risk
Maintainer bottleneck

One person owns the system; the bus factor is one.

Mitigation
Champion per BU, public RFC

System changes go through review by named champions. No silent merges.

Risk
AI builder produces non-compliant code

The generation surface drifts from the layered architecture.

Mitigation
Foundation match score gate

Score under threshold blocks PR creation. Skill v3 injects rules into the prompt.

Risk
Brand visual evolves faster than the system

Design wants to ship a new look; the system slows them down.

Mitigation
Theme layer is one file

A re-brand is a theme swap, not a code rewrite. The system accelerates re-skinning.

Risk
Out-of-band components for special cases

Teams build one-off components instead of contributing back.

Mitigation
"Promote to registry" workflow

One command lifts a local block into the registry, with audit, for re-use across the org.

Risk
Visual sameness — every screen looks the same

Strong systems can flatten product personality.

Mitigation
Themes per surface, blocks per domain

Sub-brands, dark surfaces, expressive blocks. Sameness is opt-in, not enforced.
