# Design Context — Dash Design System

## Project register

**PRODUCT** — internal app UI / dashboard tool for 10 team members at PT Dash Elektrik Indonesia. Design serves the product (operations + dispatch), not marketing.

The docs site at `https://ds.dash.com/docs` (currently localhost) is the consumer-facing surface users browse to install + reference primitives via the `dash` CLI.

## Users

**Primary**: 10 team members at Dash. Engineering background, mid-senior level, ship features end-to-end. Use the docs site to:
- Browse component catalog before generating UI
- Copy install command (`dash add button`)
- Read API + composition rules
- Verify visual + interaction spec
- Discover blocks/templates to skip composition work

**Context of use**: at desk on macOS/Linux, dual monitor, Chrome/Arc, between coding sessions. Reach docs via `cmd+K` from any tab, scan, leave. They are familiar with shadcn/ui patterns — if Dash diverges, the docs must explain.

**Secondary**: AI assistants (Claude Code, Cursor) consuming the registry through the `dash` CLI or future MCP server. They read the AI rules file (`registry/rules/dash-ai-rules.md`) and the structured registry JSON.

## Brand personality

Three words: **calm, precise, dependable**.

Voice: dry, technical, occasionally ID-lingo to match Dash domain (mitra, dispatch, tribe, surge, BMKG). Like a senior staff engineer writing internal API docs — assumes reader is smart, doesn't waste their time.

Anti-references:
- Glossy SaaS landing pages with stock photos
- Purple-gradient-on-white AI-generated marketing
- Hero sections with floating glassmorphic cards
- "Designed by AI" cliché aesthetic (neon-on-black, glow borders, sparklines as decor)

Reference brands (the *feel*, not literal copy):
- Linear app docs — quiet, precise, fast
- Notion API docs — calm + clear hierarchy
- Stripe docs — exemplary technical reference
- Vercel docs — dark + crisp, minimal chrome
- shadcn/ui docs — current baseline being matched + extended

## Aesthetic direction

**Theme**: light mode by default for the docs (engineers reading by day, OLED at night gets `class="dark"` toggle). Dark mode tested but light is the bias.

**Color strategy**: Restrained. Dash brand purple `#5e2aac` (= primary-base) as the ≤10% accent. Neutrals dominate via `bg-bg-white-0` / `bg-bg-weak-50` / `text-text-strong-950` / `text-text-sub-600`. State colors (success/info/warning/error/etc.) reserved for actual state — never decoration.

**Tone**: editorial / typographic-system. Swiss-grid feel, asymmetric splits, hairline borders. Section numerals (§ 01, § 02) on the landing. Mono labels above headings. No radial gradients. No purple-glow.

**Typography current**:
- Body: Plus Jakarta Sans (legacy from AlignUI) — *should reconsider per impeccable font selection procedure*
- Mono: ui-monospace stack
- Landing display: Instrument Serif (italic) — per impeccable rules this is on the reflex-fonts-to-reject list, **should change**

**Font selection brief words for Dash docs**: dispatch / mechanical / opinionated. Picks should feel like a 1980s Indonesian shipping logbook + Berkeley Mono + a serif with character but not Cormorant/Fraunces/Newsreader. To explore: Söhne (paid), GT Maru, ABC Diatype, Object Sans, Berkeley Mono, IBM Plex Mono replacement, or self-hosted variable like Switzer/Geist/Geist Mono.

## Design principles

1. **Quiet by default, loud where it matters.** The docs site should feel like reading a senior eng's notes, not a sales pitch. Purple appears at decision points (CTAs, active state, code highlight) — not in every card.
2. **Show the actual API.** Code blocks are real, copyable, runnable. Props tables match registry source exactly. Examples use Dash domain copy (mtr-9412, Reservasi, Halo-dash, BMKG, Lebaran rate freeze).
3. **Lateral nav for intent, not for theme.** Sidebar groups by what users want to do (Auth → Login / Register / Reset / Verification), not by internal codename (Aurora / Solaris / Phoenix).
4. **Hairline grids over heavy borders.** Cards stay light: `border-stroke-soft-200` + `shadow-custom-xs` not `shadow-custom-md`. Section dividers use the existing Divider primitive with mono labels.
5. **Motion is signal.** Animate state changes (popover entry, accordion expand, button press). Skip decorative motion. Honor `prefers-reduced-motion`.

## Constraints

- **Framework**: Next.js 16 + React 19 + Tailwind v4 + `@theme inline`
- **Token system**: `app/globals.css` defines tier (bg/text/stroke/icon × strong/sub/soft/disabled/weak/white) + state colors (success/info/warning/error/away/feature/faded × dark/base/light/lighter) + 11 color scales + shadows + radii + motion
- **Brand color is LOCKED**: `--primary-base: var(--dash-purple-500)` = `#5e2aac` (cannot change — Dash brand identity)
- **Accessibility**: WCAG AA minimum. Focus rings tone-aware (recently shipped). All interactive elements ≥44×44 touch target on mobile.
- **Performance**: docs site prerendered static at build, Shiki client-side highlight (lazy), no external font CDN — fonts self-hosted if added

## Anti-patterns to avoid (Dash-specific on top of impeccable's list)

- AlignUI codenames (Aurora / Solaris / Phoenix / Apex) visible to users — those are internal source references only
- "Pattern Library" / "Generic Templates" / similar abstract grouping labels — group by USER INTENT
- Dropdown items that look like ghost buttons (currently OK but watch for regression)
- Auth blocks using regular Button for primary CTA — must be FancyButton per source spec
