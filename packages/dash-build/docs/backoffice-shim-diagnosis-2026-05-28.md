# Backoffice Iframe HTTP 500 — Root Cause Diagnosis
Date: 2026-05-28
Status: root-cause-found

## TL;DR
`BACKOFFICE_SHIM_V2` rewrites `src/contexts/AuthContext.js` with a stub that exports `AuthProvider`, but the real backoffice app (`src/pages/_app.js:1`) imports the named binding `AuthContextProvider`. After the shim is applied that binding is `undefined`, so when `_app.js:41` renders `<AuthContextProvider>` React throws *"Element type is invalid: expected a string... mixed up default and named imports"* and the route 500s.

## Symptoms
- Error: `Element type is invalid: expected a string (for built-in components) or a class/function (for composite components) but got: undefined. You likely forgot to export your component from the file it's defined in, or you might have mixed up default and named imports.`
- When: backoffice preview iframe loads (`localhost:3101` sandbox-clone mode) after the `preview-shim apply v2 [DO NOT MERGE]` commit is in place. Crash happens at the very first render — `pages/_app.js` is the root component, so every route 500s.
- Where: cloned workspace at `~/Dash/next-backoffice-web` (or whichever path `DASH_BUILD_BACKOFFICE_PATH` points at; on this machine the source-of-truth read-only mirror lives at `/Users/irfanprimaputra.b/Work/dash/next-backoffice-web`).

## Root cause
Single-token mismatch on the auth context provider name.

- **Real source** (`src/contexts/AuthContext.js:395`): the provider that the app actually mounts is named **`AuthContextProvider`** — a wrapper that chooses between `CookieAuthProvider` and `LegacyAuthProvider` at runtime via a Firebase Remote Config flag. There is **no `AuthProvider` export** and **no default export**.
- **Shim stub** (`preview-shim.ts:150,168`): exports the provider as **`AuthProvider`** (named) and re-exports it as the module's default.

When the shim overwrites the file, `_app.js` (`import { AuthContextProvider }`) gets `undefined`, React renders `<undefined>` and throws the canonical "Element type is invalid... mixed up default and named imports" error.

This is a classic name-drift bug: the shim was written against an older or assumed export shape (`AuthProvider`) that never matched the real backoffice export (`AuthContextProvider`). The default export in the stub is what's misleading the error message — JSX never tries the default, but React's hint mentions default-vs-named because *one* of the candidate shapes (`import AuthProvider from "..."`) would have worked.

## Fix options

### Option 1: Rename the shim export to match the real binding (minimal patch)
- Change: `packages/dash-build/src/runs/preview-shim.ts` — in `BACKOFFICE_AUTH_CONTEXT_STUB`:
  - L150: `export function AuthProvider({ children }) {` → `export function AuthContextProvider({ children }) {`
  - L168: `export default AuthProvider` → `export default AuthContextProvider` (or drop the default entirely — see Option 3)
  - Bump shim `version: 2` → `version: 3` so the cherry-pick exclude filter and `verifyShimNotInBranch` still trip on the new commit subject, and old `v2` shim commits in any stale clones get superseded on the next bootstrap.
- Pros: One-token rename. No change to consumer files. Keeps the shim self-contained. Matches the established "preview parity" contract.
- Cons: Have to bump the version and remember to update any unit test that pins the literal `AuthProvider` name (search `dash-build/test` / `dash-build/src/**/*.test.ts` for the string before merging).
- Risk: low.

### Option 2: Re-export both names from the shim
- Change: keep `AuthProvider` for backwards-compat and add `export { AuthProvider as AuthContextProvider }` to the stub. Same version bump.
- Pros: Defensive — works whether a consumer imports either name. Useful if portal-v2 or any future repo lands on a third name.
- Cons: Hides the mismatch instead of naming the real contract. A second consumer might later import yet another alias and silently break again. Marginally more code in the stub.
- Risk: low.

### Option 3: Drop the default export from the stub entirely (alignment with real shape)
- Change: remove `export default AuthProvider` (L168). Combine with Option 1's rename. The real `AuthContext.js` has no default export, so the stub shouldn't either — the default is what makes the React error message specifically blame "mixed up default and named imports", which is a misleading signal during future debugging.
- Pros: Stub now exactly mirrors the real module's export surface. Future name drift gets a faithful error.
- Cons: If any *other* file (none found today) ever does `import AuthProvider from '@/contexts/AuthContext'`, it'd break — but that wouldn't compile against the real source either, so the constraint is correct.
- Risk: low.

### Recommended
**Option 1 + Option 3 combined** — rename `AuthProvider` → `AuthContextProvider` *and* drop the default export. This brings the stub's export surface to exact parity with the real module (verified via grep over the whole repo: only `{ AuthContext, AuthContextProvider, UserAuth }` are ever imported in production code). Bump `BACKOFFICE_SHIM_V2` → `BACKOFFICE_SHIM_V3` (new constant, register in `SHIMS` array at preview-shim.ts:595, keep V1/V2 exported for unit-test pinning per the file's stated convention at L307-312, L361-365).

After the patch, re-bootstrap the backoffice clone so the new `preview-shim apply v3 [DO NOT MERGE]` commit replaces the broken v2 commit; the iframe should then render past `_app.js`.
